# RTSP 渐进式迁移与完善计划总索引

- 状态：已完成
- 目标：以渐进式方式移植 `vendor-ref/rtsp-rs` 的测试与相关实现，补齐本仓库 `cheetah-rtsp-core`、`cheetah-rtsp-driver-tokio`、`cheetah-rtsp-module` 的 RTSP 主线能力。
- 方法：每次只处理一个 vendor 测试用例，先完整复制测试，再根据该测试反推并移植最小实现。
- 完成判定：所有阶段计划状态改为“已完成”，且对应测试全部能在本地仓库中运行通过。

## 总体约束

- 严格遵守 `core + driver + module` 三段式，不复制 vendor 的 crate 结构。
- `cheetah-rtsp-core` 只承接纯 Sans-I/O 协议原语与显式输入输出模型。
- `cheetah-rtsp-driver-tokio` 只承接 Tokio I/O、分段收发、混合 RTSP/interleaved 数据驱动与 backpressure。
- `cheetah-rtsp-module` 只承接引擎接入、会话编排、鉴权、资源分配、业务状态与端到端流程。
- vendor 中围绕客户端连接的语义，统一采用“服务端映射”方案：
  - 纯解析、限制、状态转换语义下沉到 `cheetah-rtsp-core`。
  - 混合包、连接生命周期、缓冲与关闭行为落到 `cheetah-rtsp-driver-tokio`。
  - PLAY / RECORD / PAUSE / TEARDOWN / keepalive 等业务语义由 `cheetah-rtsp-module` 承接。
- 所有从 vendor 迁入的日文注释必须同步翻译为中文，不允许延期到最后统一处理。

## 迁移顺序

1. 先建立测试承载与文件拆分边界，避免后续继续把逻辑堆入 `module.rs` 与 `keepalive.rs`。
2. 再迁移 `pbt_http`、`pbt_limits`、`fuzz_http_*`、`fuzz_rtsp_limits`，先把消息增量解码与限制模型打稳。
3. 然后迁移 RTP/RTCP 与 RTSP 工具原语，保证 module 不再维护私有解析逻辑。
4. 再迁移 SDP 通用能力，把 `module` 里的 SDP 业务映射建立在 core 原语之上。
5. 最后用 driver/module 集成测试承接 vendor 客户端连接语义和本地主线业务场景。
